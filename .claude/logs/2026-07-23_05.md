## Judgment calls (disclosed)

- **Sticky context (item 3)** realized as the league selector living in the persistent sidebar (moving Home↔Dashboard↔Draft never makes you re-pick the league; the league is in the URL path). Deliberately did NOT build a cross-section client store for week/team — week is a dashboard-local `?week=` param, "selected team" isn't a dashboard concept (only the draft board's client-only selfRosterId), and a global store would fight the shipped URL-driven RSC design and the LeagueSelector's deliberate param-drop. "where applicable" in the item covers this: league is the applicable sticky context, and the friction the item targets (re-selecting on section change) is eliminated.
- **Command-center cards = only what exists today** (standings + power snapshots + a draft deep-link), per the item's own "standings/power-rank snapshot now; luck/waiver/playoff/trade once Wave 5 exists." Did NOT add placeholder Wave-5 cards on the home — the future slots live in the sidebar (item 4), keeping the two surfaces distinct.
- **Future slots**: wave numbers kept in the `FUTURE_NAV` config (the code-level mount point that satisfies "leave explicit route slots") but not rendered in the UI — a "Soon" tag is cleaner than leaking internal wave numbering to the user.
- **`login/actions.ts` left unchanged** — post-sign-in still lands on the first league's draft board, a SEPARATE Nick-signed decision (still a valid route). It's now inconsistent with root auto-land (home), but changing a signed decision is outside this fold's four items and not a correctness bug — flagged for Nick in the report / nick_pending rather than changed unilaterally.
- **Layout-level `listConnectedLeagues` await isn't covered by loading.tsx** (Next wraps the page, not the sibling layout, in the Suspense boundary) — accepted: one lightweight indexed query, and the layout persists across section navigation so it runs once per league, not per nav.
