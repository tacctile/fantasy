## Clarify (AskUserQuestion — 3 questions, all Nick-signed)

The one real design decision was the cross-component gap: the badge must sit near the position filter (toolbar) but the tier-depth signal to pair it with (topTierSize) lives in the BPA sidebar panel's fetch.

1. **Data flow → LIFT tiers up via callback.** BPA panel reports its already-computed `context.tiers` up to the shell via a new optional `onTiers`; shell computes runs client-side from `livePicks` and passes both to the toolbar badge. Reuses the ONE existing summarizeTiers result (no second tier compute); degrades to run-only when the sidebar is hidden below `lg`. Rejected: run-flag-only (loses the demand-vs-panic signal Nick asked for); badge inside the panel (contradicts "near the position filter").
2. **Content → chip + tooltip.** Muted pill (`PositionBadge` + "run"); tooltip carries "N POS in the last W picks · M left in the best available tier". Rejected: adaptive inline wording (more prominent than the non-blocking posture wants); dot-only (loses the at-a-glance position/word).
3. **Empty/multiple → appear-only, one per run.** Nothing rendered when no run is active (no reserved chrome); one badge per active run in the pure layer's count-desc→position-asc order. Rejected: always-present "No runs" area (persistent chrome for a mostly-empty state).
