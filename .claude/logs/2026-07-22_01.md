## Clarify decisions (Nick, via AskUserQuestion)

1. **Composite PK `(league_id, native_roster_id)`** — not a surrogate uuid. Matches the wiki pair-identity decision literally; downstream tables carrying the composite FK get `league_id` automatically (Schema Rule #1 for free); ingestion writes arrive already keyed by roster_id/teamId, no uuid lookup.
2. **`native_roster_id integer`** — Sleeper `roster_id` and ESPN `teamId` are both wire integers; integer IS as-received. The "IDs as strings" rule exists for Sleeper's large numeric-string IDs and D/ST abbreviations, neither of which applies here.
3. **Owner metadata as first-class columns** — `owner_native_id text` nullable (Sleeper `owner_id` / ESPN `primaryOwner`) + `co_owner_native_ids text[] not null default '{}'` (Sleeper `co_owners` / ESPN non-primary `owners[]`). Text because Sleeper user_id is a large numeric string and ESPN memberId a GUID.
4. **Display columns now** — nullable `team_name` + `owner_display_name`, populated by Wave 2 sync (Sleeper team names come from the users endpoint fallback hierarchy, not the rosters endpoint; ESPN's from the team object).
