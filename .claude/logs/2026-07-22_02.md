## Clarify decisions (Nick — all recommended options)

1. **Slot model: normalized enum + native detail.** `slot` Postgres enum (`starter`/`bench`/`reserve`/`taxi`) as the cross-platform role, plus nullable `starter_slot_index` (Sleeper: index into `starters[]`) and `espn_lineup_slot_id` (ESPN: raw `lineupSlotId`). Bench derivation happens once at Wave 2 ingestion per the wiki's centralize-it decision; native columns preserve slot-mapping fidelity.
2. **ESPN acquisition metadata: leave out.** No registered v1 feature consumes `acquisitionType`/`acquisitionDate` (waiver/FAAB analysis reads the transactions endpoint). Add via new migration later if a wave needs it.
3. **Per-league player uniqueness: enforce.** `unique (league_id, sleeper_player_id)` on top of the composite PK — a player on two rosters in one league snapshot is corrupt data; fail loudly at sync time rather than double-count silently in free-agent/availability derivations. The wiki's transient-overlap open question concerns arrays within ONE roster, untouched by this constraint.
