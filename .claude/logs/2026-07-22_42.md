## Item 9 — recompute on any pick, any source → [x]

**Shipped:** the panel's fetch `useEffect` keys on `[leagueId, livePicks, selfRosterId]`. livePicks is the shell-owned confirmed snapshot (the SAME one the board reads — never a second pick-event feed). The shell's handlePicks fingerprint bail-out means an identical snapshot keeps the same array reference, so a steady 5s poll returning no new picks does NOT re-run the effect — only a genuine change (manual pick, Sleeper poll, undo, optimistic adopt) flips the reference and refetches. The shrinking-pool recompute itself lives in getBpaRecommendations (pool = projections − draft_state), not cached here. A monotonic request-seq ref drops out-of-order responses (a slower earlier pick can't clobber a newer result). Stale-while-revalidate: state stays 'ready' across refetches (only the first load shows loading), so picks landing every few seconds never flicker the panel empty. selfRosterId change also refetches (need signal).

## Judgment calls (disclosed)

- Stale-while-revalidate with NO separate "refreshing" spinner: keeps the panel from flickering empty on every pick, and avoids a synchronous setState in the effect body (react-hooks/set-state-in-effect, active in this repo). The request-seq guard covers correctness; a refresh indicator was deemed polish and omitted.
- Panel fetches always (useful pre-draft as a draft assistant), independent of session-active; only the per-row Draft control gates on draftEnabled — same as the main table.
- The panel shows name/pos/team/ADP/VORP + need badge, not projectedPoints/replacementPoints — the build-file item names ADP + base VORP + need; projection internals stay in the service.
- 'starter' need badge borrows --warning (RosterPanel's actionable-need color) rather than a new token — one "this addresses a roster need" color across the surface; teal stays reserved.
- DraftablePlayer added rather than casting a BpaRecommendation to DraftBoardPlayer — typed, no `any`, smallest honest seam for the shared handler.
