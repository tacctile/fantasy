## The prolabel discovery (major)

First `db push` refused: 23 remote migration versions absent locally. Investigation via read-only Supabase connector calls: linked project `tszssadgsxjoymcttlwd` is literally named **prolabel** — Nick's shared multi-app catch-all with ~49 live tables (todd chat, elliott pricing, trip planner, loan/valuation), 23 foreign history rows in `supabase_migrations.schema_migrations`, and 2 pre-existing auth users. Org `tacctile` has one other project (`ghc_crm`); a new project would cost $0/month (verified via cost API).

Nick asked for the mechanical blast radius of `supabase migration repair --status reverted` before deciding; answered: it DELETEs exactly the listed version rows from the shared ledger (those rows ARE the other apps' records), never executes DDL, zero runtime impact on live apps, but erases the shared record that those migrations ran. His directive: stay in prolabel, and use a path that touches none of the 23 entries.

**Solution adopted: empty stub migrations.** 23 `*_prolabel_shared_history_stub.sql` files (comment-only) mirror the foreign versions locally; `db push` then only pushes fantasy's own migrations (proven by `--dry-run` before the real push) and only ever INSERTs fantasy's own history rows. Standing rules recorded in ARCHITECTURE.md ("Shared-Database Constraints") + MANUAL_SETUP_CHECKLIST decision of record: never `db reset`, never `migration repair`; new foreign versions → new stubs; RLS owner policies must match Nick's specific `auth.uid()` (shared auth namespace); check table-name collisions against prolabel before new table families.

## Addendum — provenance self-audit and corrective migration (same session, post-report)

Nick asked for a full provenance audit of every decision. Honest finding: the crosswalk
and enum were genuinely wiki-driven; the `players` column decomposition came from general
training knowledge of Sleeper's API without consulting the sleeper-api category, despite
one unspent page of the 3-page budget and ROUTING.md mapping "Sleeper endpoint structure"
to that category. Flagged to Nick along with all other general-knowledge decisions
(timestamps/trigger, cascade behavior, metadata jsonb default, RLS-at-creation, migration
naming, all prolabel ops work — the latter legitimately outside wiki scope).

Nick authorized verification: read `sleeper-api/players-endpoint.md` (page 3, within
standard budget) and `sleeper-api/player-data-quirks.md` (page 4, explicitly authorized
by prompt per ROUTING.md's "prompt explicitly allows more").

Audit outcome: no field-name mismatches; `full_name` nullable confirmed correct for D/ST;
`status` as text (not enum) validated (value set unpublished/unstable); absence of
`bye_week` validated (schedule-derived by design). Five fields had wiki-assigned platform
roles and were promoted from `metadata` to first-class columns via corrective migration
`20260722042128_add_players_sleeper_reference_fields.sql` (created via `supabase migration
new`, dry-run-verified, pushed by Nick due to the classifier block, verified live):
`fantasy_positions text[]` + `search_rank integer` (the wiki's four-field relevance filter
with status/team), `injury_status text` (two-field availability rule — never conflate with
`status`), `search_full_name text` + `birth_date date` (last-resort flagged name-matching
key per the identity ADR). Deliberately not added: bye_week, practice-squad/elevation flag
(elevation invisible to the daily dump; Wave 2 sources it fresher), depth-chart fields
(no wiki-assigned role; frequently null/stale for fringe players — stay in metadata).

Lesson recorded for future sessions: when a build item's own wording forces inventing
concrete field names, that is the "item genuinely needs more" trigger for extra wiki
pages — the category exists precisely for that moment.
