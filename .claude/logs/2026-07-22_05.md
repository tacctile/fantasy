## Clarify step (4 questions, all answered before any code — Nick took all four recommended options)

1. **PK composition** — item text lists 4-column scoping incl. season_year; matchups precedent excludes it from the PK. Nick chose **PK (league_id, sleeper_player_id, week)** with `season_year` duplicated as a non-PK column — a leagues row is already per-season, so season_year in the PK would be redundant.
2. **Roster attribution** — Nick chose **native_roster_id + was_starter**: full per-week attribution. This is the only table that can hold historical per-week lineup facts (roster_players is current-snapshot; mRoster can't reconstruct history per the wiki), and Wave 5's lucky/unlucky + positional breakdowns are registered consumers. Every row is inherently a rostered player (free-agent scores never appear in these payloads), so `native_roster_id` is NOT NULL with composite FK → rosters.
3. **Score columns** — Nick chose **single `points numeric not null`**: a row only exists when the source reported a numeric value; zero lands as-received (zero-vs-not-played is a consumer concern); negatives legitimate (DST/turnovers). ESPN projected/live excluded — field names are a declared wiki open question; Wave 2 adds them via new migration once verified live.
4. **Finality** — Nick chose **fetched_at + is_final mirroring matchups exactly**: same payloads, same correction-window reality (DST/IDP most volatile), so Wave 2's promotion policy treats both tables uniformly. `fetched_at` nullable per the matchups precedent.
