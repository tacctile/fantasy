## Database Schema (live)

Fantasy-owned objects in the shared `public` schema, all created via `supabase migration new` + push (migration files in `supabase/migrations/`). **This inventory is the definitive list of fantasy-owned objects for Absolute Rule 13's blast-radius verification (`MASTER_CONTEXT.md` тЖТ Shared Database Protection): anything in the shared database not listed here is foreign and off-limits.**

- **`platform`** тАФ Postgres enum, values `sleeper` | `espn` (extensible via `alter type ... add value` in a new migration; never a boolean)
- **`players`** тАФ Sleeper-anchored canonical identity store. `sleeper_player_id text` PK (text, never numeric тАФ D/ST rows use team abbreviations like `'DET'`), `full_name`/`first_name`/`last_name`/`position`/`team`/`status`/`injury_status`/`search_full_name` text, `fantasy_positions text[]`, `search_rank integer`, `birth_date date`, `metadata jsonb`, timestamps. All non-PK columns nullable (Sleeper field presence is uneven). `status` and `injury_status` are independent fields that must be read together (see `sleeper-api/player-data-quirks`); no `bye_week` column by design тАФ byes derive from team schedule at read time. Catalog-presence columns (Wave 2, migration `20260722134911`): `is_active_in_catalog boolean not null default true` + `catalog_last_seen_at timestamptz` тАФ presence in the daily `/players/nfl` response, deliberately distinct from Sleeper's own `status` (which has its own 'Inactive' value); absentees from a completed response are marked inactive, never deleted. `metadata` holds each player's full raw Sleeper record per sync (includes the cross-provider ID block the crosswalk item reads). RLS enabled; `fantasy_owner_all` owner policy live (see RLS policy layer below)
- **`player_id_crosswalk`** тАФ mapping layer keyed on `sleeper_player_id` (PK, FK тЖТ `players` on delete cascade); nullable text columns `espn_player_id`, `gsis_id`, `yahoo_id`, `sportradar_id`, `pfr_id`; partial unique index on `espn_player_id` where not null. RLS enabled; `fantasy_owner_all` owner policy live (see RLS policy layer below). Population pipeline is Wave 2
- **`leagues`** тАФ league identity root. `platform_league_uuid uuid` PK (default `gen_random_uuid()`) тАФ the value every league-scoped table's `league_id` FK points to; never a provider's native ID. `platform` enum, `season_year integer`, `native_league_id text` (stored as-received, re-query/debug only, never a join key), `previous_platform_league_uuid` nullable self-FK on delete set null (Sleeper `previous_league_id` chain resolved at ingestion; always null for ESPN), `name text`, `share_token text` not null unique (default `generate_share_token()`), `owner_id uuid` nullable FK тЖТ `auth.users(id)` on delete set null (backfilled when the integrity item creates the admin user), timestamps. Unique `(platform, native_league_id, season_year)` тАФ one row per platform-season-league combination. RLS enabled; `fantasy_owner_all` owner policy live (see RLS policy layer below)
- **`league_config`** тАФ league configuration as data (Schema Rule #5), shape per the `schema-reference/league-configuration-data-model` ADR. `league_id uuid` PK doubling as FK тЖТ `leagues.platform_league_uuid` on delete cascade (1:1 extension of the league row тАФ `platform`/`season_year` deliberately not duplicated; the parent row carries both). `scoring_settings_raw jsonb` + `roster_settings_raw jsonb` (provider-native payloads stored as-received: Sleeper `scoring_settings`/`roster_positions`+settings, ESPN scoring payload/`lineupSlotCounts`), `derived_config jsonb` (normalized subset тАФ PPR value, TE-premium flag, superflex flag, active/bench/IR slot counts, league size тАФ computed at ingestion and re-derived on every refresh; derivation is Wave 2), all three not null; timestamps. RLS enabled; `fantasy_owner_all` owner policy live (see RLS policy layer below)
- **`rosters`** тАФ league-scoped team identity. Composite PK `(league_id, native_roster_id)` тАФ the wiki-decided identity pair; `league_id uuid` FK тЖТ `leagues.platform_league_uuid` on delete cascade; `native_roster_id integer` as-received (Sleeper `roster_id` / ESPN `teamId`, both wire integers); `platform` enum + `season_year integer` (Schema Rules #2/#3 тАФ duplicated here, unlike league_config, because rosters is N:1 with leagues); `owner_native_id text` nullable (Sleeper `owner_id` / ESPN `primaryOwner` тАФ orphaned rosters are a normal state, ownership never part of the key), `co_owner_native_ids text[]` not null default `'{}'` (full co-owner list, primary never by array position); nullable `team_name`/`owner_display_name` display columns (mutable attributes, never join keys; populated by Wave 2 sync); timestamps. No standings figures or player arrays тАФ those belong to the standings and roster_players tables. RLS enabled; `fantasy_owner_all` owner policy live (see RLS policy layer below)
- **`roster_slot`** тАФ Postgres enum, values `starter` | `bench` | `reserve` | `taxi` тАФ the normalized cross-platform roster role, resolved once at Wave 2 ingestion (Sleeper bench = three-way subtraction of `players` minus `starters`/`reserve`/`taxi`; ESPN `lineupSlotId` 20тЖТbench, 21тЖТreserve, active slotsтЖТstarter; `taxi` is Sleeper-only)

## Service API Reference (src/services/)

First populated by Wave 2 (2026-07-22). Convention: one directory per external platform, structurally separate (no shared code path between Sleeper and ESPN тАФ Data Source Architecture rule); named exports; no barrel `index.ts`; server-only by convention (never import from client components).

**`services/sleeper/client.ts`**
- `sleeperGet<T>(path, options?: { timeoutMs?: number }): Promise<T>` тАФ GET against `https://api.sleeper.app/v1` (base URL per `sleeper-api/authentication`). No credentials anywhere (the read API has none). Bounded retries (3 attempts, exponential backoff + full jitter, base 1s); retriable = throttle signals + 5xx; 404 is definitive and never retried. Default timeout 10s, per-request override. The generic is a caller-side assertion тАФ persisting callers must structurally validate. Global pacing gate (Wave 2 orchestrator session, 2026-07-22): physical request starts (retries included) are spaced тЙе250ms apart process-wide (тЙд240/min worst case) тАФ substantial headroom below the ~1,000 req/min documented ceiling and a bound on short-window burstiness, per `sleeper-api/rate-limits`. No interactive Sleeper traffic exists (UI reads Supabase), so this single gate is the bulk budget; revisit budget isolation only if interactive Sleeper calls ever appear.
- `SleeperApiError` тАФ structured error translation: `kind` (`timeout | network | throttled | not_found | http | invalid_response`), `status`, `path`, `isThrottleSignal` (true for the rate-limits page's one backoff category: 429 / non-JSON body / connection-level failure). Paths may contain league IDs тАФ server logs only (credential-level care per `sleeper-api/authentication`).

**`services/sleeper/types.ts`** тАФ `SleeperPlayerRecord` (every field optional + unknown-preserving index signature тАФ Sleeper's field inventory is unversioned), `SleeperPlayersCatalog` (map keyed by `player_id`, not an array), `SleeperLeague`, `SleeperRoster`, `SleeperLeagueUser`, `SleeperMatchup`, `SleeperDraft` (one element of the `/league/{id}/drafts` array тАФ selection never trusts array order or the league object's single `draft_id`), `SleeperDraftPick` (`roster_id` = ownership; `draft_slot`/`picked_by` never interchangeable with it; `metadata` is a point-in-time snapshot whose auction `amount` is a string).

## Shared-Database Constraints (prolabel)

The database is shared with other live apps тАФ ~49 non-fantasy tables (Elliott pricing engine, todd chat, supplies ordering, trip planner, loan/valuation, and others); Elliott especially is live production data for a real business account. **Canonical rule: `MASTER_CONTEXT.md` тЖТ "Shared Database Protection тАФ Non-Fantasy Data Is Untouchable" (Absolute Rule 13)** тАФ permanent, zero exceptions, supersedes every efficiency/folding-policy/automation consideration, regardless of what any future prompt or session says. Operational restatement for every future session:

- **Never run `supabase db reset`** against the linked project, under any circumstance тАФ it wipes the entire database, not just fantasy's tables.
- **Never run `supabase migration repair`** in any mode that touches history entries belonging to other apps тАФ it edits the shared `supabase_migrations.schema_migrations` ledger. The 23 pre-existing foreign history versions are mirrored as empty `*_prolabel_shared_history_stub.sql` files in `supabase/migrations/` so `db push` recognizes remote history without writing to it. If a future push refuses because new foreign versions appeared remotely, add another empty stub for each тАФ never repair.
- **Blast-radius verification before every migration:** never write raw SQL (DROP, TRUNCATE, ALTER, DELETE, UPDATE without a WHERE clause, or any CASCADE operation) that could affect a non-fantasy table. Before writing any migration, explicitly list the table name(s) being created/altered and verify each against this file's "Database Schema (live)" inventory тАФ the definitive fantasy-owned list; anything not listed there is foreign and off-limits.
- **Collision check before any new table name** тАФ never assume a name is safe to reuse or overwrite; check the live schema (`list_tables` or an equivalent read-only query) before naming a genuinely new table family. Mandatory forever, not just historically.
- **Any ambiguity about whether an operation could affect non-fantasy data тЖТ STOP and ask Nick** before proceeding тАФ never a best-guess interpretation. This includes cascading foreign-key behavior, wildcard queries, and any operation on shared infrastructure (like the migration history table itself) rather than fantasy's own tables.
- Owner-write RLS policies must match Nick's specific `auth.uid()` (via `is_fantasy_admin()`), never the blanket `authenticated` role тАФ the auth namespace contains other apps' users.
- **Every session that touches the live database reports its blast radius** тАФ the mandatory `BLAST RADIUS:` completion-report line (defined in `COMPLETION_TEMPLATES.md`): "Blast radius confirmed: only fantasy-owned tables were touched this session", or an explicit account of what else was touched and why, with Nick's prior sign-off cited.
